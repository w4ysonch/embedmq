# embedmq — 嵌入式 Linux & RTOS 轻量消息分发库

## 项目简介

**embedmq** 是一个零依赖的 C 语言消息分发库，专为嵌入式 Linux 和 RTOS 设计。核心机制是 UUID dispatch + 环形缓冲 + 信号量唤醒，实现线程间异步通信。

**作者**：chenhuisheng1  
**语言**：C（C11），附可选 C++ 封装  
**证书**：MIT

---

## 解决什么问题

嵌入式 Linux / RTOS 项目需要线程间消息传递，但现有方案要么太重（DBus 需要 daemon + glib），要么太简陋（手写队列线程不安全、没有统一 dispatch）。

| | embedmq | DBus | ZeroMQ | 手写队列 |
|---|---|---|---|---|
| 依赖 | 零依赖 | daemon + glib | libzmq | 无 |
| 平台 | Linux / RTOS / 裸机 | 仅 Linux | 全平台 | 全平台 |
| 零堆分配 | ✅ | ❌ | ❌ | 手动 |
| API 复杂度 | 3 个函数 | 极复杂 | 中等 | — |
| UUID dispatch | ✅ 快 | ❌ 字符串 | ❌ | — |

## 核心设计

三个组件：

### 1. Registry（注册表）
注册时把 `name + format` hash 成 `uint32_t UUID`，运行时 dispatch 只比较整数，不做字符串匹配。

```c
embedmq_register(q, "battery.changed", on_battery_changed, user_data);
```

### 2. Ring Queue（环形缓冲）
用 `atomic_int head` 和 `atomic_int tail` 实现的环形缓冲区，支持两种模式：
- **动态模式**：内部 malloc 分配缓冲区
- **静态模式**：调用方提供 buffer（零 malloc，适合裸机）

队列中的消息格式：
```
[ uint32_t uuid ][ uint16_t length ][ payload bytes ]
```

### 3. Dispatcher（消费线程）
内部起一个消费线程跑 `while(1)`，在信号量上阻塞等待消息。有消息时被唤醒，排空队列，按 UUID 查表调用对应 handler。

```
生产者线程：
  embedmq_post(q, "battery.changed", &info, sizeof(info))
    → pack UUID + payload 进环形缓冲
    → semaphore_give() 唤醒消费者

消费者线程（内部）：
  semaphore_wait()  ← 队列空时阻塞在这里
    → 读 UUID
    → 查注册表找 handler
    → 调用 handler(data, size, user_data)
```

## 目标 API

```c
// 创建 dispatcher（内部起消费线程）
embedmq_t* embedmq_create(const embedmq_config_t* config);

// 注册 handler
int embedmq_register(embedmq_t* q, const char* name, embedmq_handler_fn fn, void* ctx);

// 投递事件（非阻塞，线程安全）
int embedmq_post(embedmq_t* q, const char* name, const void* data, size_t size);

// 销毁
void embedmq_destroy(embedmq_t* q);
```

Handler 签名：
```c
typedef void (*embedmq_handler_fn)(const void* data, size_t size, void* ctx);
```

静态模式（零 malloc）：
```c
// 调用方提供 buffer，内部不 malloc
embedmq_t* embedmq_create_static(uint8_t* buf, size_t buf_size, const embedmq_config_t* config);
```

## 平台抽象层（PAL）

为了同时支持 Linux（pthread）和 FreeRTOS / 裸机，所有 OS 原语走 PAL：

```
pal/linux/embedmq_pal.c    → pthread_mutex, pthread_cond, pthread_create
pal/freertos/embedmq_pal.c → xSemaphore, xTaskCreate
pal/none/embedmq_pal.c     → 轮询 fallback（无 OS）
```

## 关键设计决策

**为什么用 UUID 而不是字符串 dispatch？**
注册时一次性 hash，运行时只比整数，热路径零字符串操作。对嵌入式设备每个 cycle 都重要。

**为什么支持零 malloc？**
嵌入式设备（MCU 和受限 Linux）要避免堆碎片和不确定延迟。静态模式让调用方提供固定大小 buffer，生命周期完全可控。

**为什么用信号量而不是忙等？**
忙等（spin-wait）白白浪费 CPU。消费线程阻塞在信号量上，只有生产者 signal 时才醒，和 FreeRTOS queue 及 AIPL 的 BLLThreadLoop 是同一个思路。

**为什么不用 POSIX 消息队列（mq_open）？**
POSIX mqueue 是跨进程的，有内核开销，创建时要确定大小和数量上限。embedmq 是进程内线程间通信，更轻，运行时可配置。
